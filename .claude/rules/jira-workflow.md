# JIRAワークフロー

単一の真実はJIRAのNSプロジェクト（hittslabs.atlassian.net）。作業の状態はすべてチケットに残す。

## 役割分担（重要）

- **人間が作るのはEPICだけ**（ゴール・受け入れ基準（Acceptance Criteria）・検証方法を含む）。EPICは変更しない・待たない・催促しない
- **ストーリー・タスクはエージェントが起票する。** 人間の起票を待たずに、EPICの受け入れ基準から必要なストーリーを自分で切り出すこと。分割粒度・順序はあなたの裁量
- **バグの起票者はエージェント・Verifier・人間のいずれでもよい**（発見した者が起票する）。扱いはすべて同じ：EPIC配下にぶら下げ、ループが消化する
- 人間の関与は「EPICの作成・needs-humanへの回答・PRマージ・EPICの受け入れ」のみ

## ワークフロー遷移表

ステータス：`未着手 → 進行中 → レビュー中 → 完了`

### EPIC（ループの**外**。人間とループの境界信号）

EPICのステータスの意味：「進行中」＝ループに委任中／「レビュー中」＝人間の番。

**EPICのステータス遷移はすべて人間のみが行う。エージェント（実装ループ・Verifier・Routine）はEPICのステータスを一切変更しない。** EPICの遷移はループの起動トリガーであり、エージェントが触ると自己発火事故を起こすため（2026-07-12に発生）。エージェントがEPICに対してできるのはコメントとラベル（`needs-human`）のみ。

| 遷移 | 実行者 | 条件 |
|---|---|---|
| 起票（未着手） | **人間のみ** | ゴール（停止条件）・受け入れ基準・検証方法が記載済み |
| 未着手 → 進行中 | **人間のみ**（またはfire API） | ＝**ループ起動トリガー** |
| 進行中 → レビュー中 | **人間のみ** | ループ正常終了の合図（全子チケットがレビュー中以上 ＋ **EPIC受け入れ検証（完全モード）PASS報告がEPICにコメント済み**）を人間が確認して遷移 |
| レビュー中 → 完了 | **人間のみ** | 最終PRマージ済み ＋ 全子チケット完了 ＋ 人間の受け入れ判断 |
| レビュー中 → 進行中 | **人間のみ** | 差し戻し（＝ループ再着火の合図） |
| （途中停止） | — | ステータスは変えず `needs-human` ラベルで表現。解除＝再開の合図 |

### ストーリー／タスク／バグ（ループの**内**。エージェントが管理）

| 遷移 | 実行者 | 条件 |
|---|---|---|
| 起票（未着手） | エージェント（バグは人間・Verifierも可） | 親EPIC必須・**受け入れ条件**記載（テンプレ準拠） |
| 未着手 → 進行中 | エージェント | 着手時（1枚ずつ）。着手コメントを残す |
| 進行中 → レビュー中 | エージェント | **PR作成 ＋ Verifierストーリー検証（軽量）PASS ＋ 検証記録ページ（Confluence）のURLをチケットにコメント済み**。DEFERRED項目がある場合はPRとチケットに明記 |
| レビュー中 → 完了 | エージェント | **人間がPRをマージしたことを確認後**（マージ前に完了にしない） |
| レビュー中 → 進行中 | エージェント | 人間の差し戻しコメント・Verifier FAILへの対応で再着手するとき |
| （保留） | エージェント | 同一チケット3回連続失敗→ステータス維持のまま `deferred` ラベル＋保留コメント。**ループは停止せず**次のチケットへ（再着手は人間の合図後） |

**原則**：ステータス遷移は常に**事実の後追い**。「もうすぐマージされるから完了」「PASSするはずだからレビュー中」のような先回り遷移は禁止。

## 起票ルール

- ループ起動（EPICが「進行中」遷移、またはAPI呼び出し）したら、まず受け入れ基準を読んでストーリー一式を起票してから実装に入る（着手後の追加起票も可）
- ストーリー・タスクは対象EPICの子として起票する。親のないチケットを作らない
- バグは発見時に該当EPIC配下へ起票するだけ。その場で修正しない（割り込み禁止）。修正PRには回帰テスト必須
- 1チケット＝1事象。まとめ起票しない
- 起票時の必須項目：事象/目的、再現手順または作業内容、**受け入れ条件**（そのチケット固有の完了判定。親EPICのどの受け入れ基準に対応するかも明記）

## ラベル規約

- `needs-human`：人間の判断がないと**前に進めない**事項（仕様齟齬・受け入れ基準の曖昧さ等）。ループ停止を伴う
- `deferred`：同一チケット3回連続失敗による**保留**。ループは停止せず他のチケットを継続し、未消化チケットが尽きた時点で保留一覧をEPICにコメントして停止する（Verifierの検証項目判定「DEFERRED（完全検証送り）」とは別物）
- `verifier`：Verifierが起票したバグ
- `test-quality`：テスト品質の指摘（FAIL要因ではない）

## 人間からの指示の扱い

- 人間からの指示の正式なチャネルは**JIRA**（チケットコメント・ラベル・ステータス遷移）。ループは状態観察でそれらを読む
- セッションチャットやfireのtextで直接指示を受けた場合は、**実行前に指示内容を対象チケットへコメントで転記する**（口頭指示も台帳に載せる。転記しない指示は次のセッションに存在しないのと同じ）
- チケットに残っている指示とチャットの指示が矛盾する場合はJIRAを正とし、矛盾を指摘して確認する

## コメント規約

- 作業ログは「やったこと／わかったこと／次にやること」の3点で簡潔に
- 中断時（リミット到達等）は必ず進捗コメントを残してから止まる
- 判断に迷った内容は推測で進めず、チケットに選択肢を書いて `needs-human`
