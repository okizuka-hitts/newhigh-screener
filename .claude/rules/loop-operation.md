# ループ運用

このリポジトリはEPIC駆動ループで開発する。人間は都度指示を出さない。あなたの行動はこのルールとEPICの受け入れ基準で決まる。

## 定義：EPIC駆動ループとは

> 人間が停止条件＝**受け入れ基準（Acceptance Criteria）**を書いたEPICを起点に、AIエージェントが「状態観察 → 計画（ストーリー起票）→ 実装 → 独立検証」を、Verifierの受け入れ検証で受け入れ基準を全件PASSするまで自律的に反復する開発ループ。

### 用語の区別（重要）

- **受け入れ基準**（＝ループ停止条件）：EPICに**人間が**書く。ループ全体の終了判定。充足の判定はVerifierだけが行う。「AC」という略語は使わない（ドキュメント・チケット・コメントすべてで「受け入れ基準」と書く）
- **受け入れ条件**：ストーリー・タスク・バグの各チケットに**エージェントが**起票時に書く、そのチケット1枚の完了判定。ループの1周を閉じる条件であり、ループ自体は止めない

構成要素の対応：

| 要素 | このプロジェクトでの実体 |
|------|--------------------------|
| ループの実行単位 | EPIC（1 EPIC = 1ループ） |
| トリガー | EPICが「進行中」へ遷移する、またはRoutineのAPI呼び出し |
| 状態（外部記憶） | EPIC配下のチケットツリー＋Git＋Confluence検証記録 |
| 1周（イテレーション） | 状態観察 → 次の一手を1つ選ぶ → 実装 → 検証 |
| 検証ゲート | Verifierサブエージェント（実装と分離、独立判定） |
| 停止条件（正常） | EPIC受け入れ検証（完全モード）で受け入れ基準を全件PASS → PASS報告をEPICにコメントして停止（EPICの遷移は人間のみ） |
| 停止条件（異常） | 仕様齟齬・予算リミット → `needs-human`（3回失敗のチケットは `deferred` で保留し、ループは継続） |
| 人間の関与点 | 入口（EPICと受け入れ基準を書く）／PRレビュー・マージ／受信箱（needs-human）／出口（受け入れ判断） |

「ループ」と言うとき、それは会話のターンでもセッションでもなく、**この表の単位**を指す。

## ループの1周

1. **状態観察**：対象EPIC配下の未完了チケットを確認する（進行中→積み残しバグ→新規ストーリーの優先順）
2. **次の一手を1つ選ぶ**：1チケット＝1作業単位。並行して複数チケットを進めない
3. **実装**：ブランチを切り、実装し、テストを書き、PRを作る
4. **検証**：完了を主張する前に必ず `verifier` サブエージェントを**ストーリー検証（軽量モード）**で起動する。verifierのPASSなしに完了を主張しない。確認項目と結果はVerifierがConfluenceの検証記録ページに残し、チケットへリンクする（**記録のないPASSは無効**）
5. **マージ待ち**：**PRのマージは人間だけが行う。** VerifierのPASS結果をPRとチケットにコメントし、チケットを「レビュー中」へ遷移したら、マージを待たずに次のチケットに進んでよい（人間のレビューはループをブロックしない）。マージで差し戻しコメントが付いたら、そのチケットを再度取り上げる

## ループの継続（重要）

**EPIC配下の子チケットがすべて「完了」になるまで、EPICの実装ループを停止しない。** 未消化のチケット（未着手・進行中・差し戻し。`deferred` 保留中は除く）が残っている限り、1周ごとに区切って止まらず、続けて次の一手を取る。「1チケット終えたので今回はここまで」という自己判断の中断は禁止。停止してよいのは「停止条件」に該当したときだけ。

マージ待ちのみが残る状態（全子チケットがレビュー中以上）では正常終了の手順まで進めて停止してよいが、**人間のマージ後に子チケットを「完了」へ遷移させるところまでがループの責務**であり、全子チケットが「完了」になるまでEPICのループは終了扱いにならない（セッションをまたぐ場合は再開時の状態観察で未処理の完了遷移を必ず処理する）。

## 検証とマージの順序（重要）

**EPIC受け入れ検証（完全モード）は、EPICの最終PRを人間のマージ判断に出す前に、対象ブランチ上で実施しPASSさせる。** 「マージ後に完全検証する」は禁止。人間がマージを判断する時点で、Verifierの完全検証報告（実APIのE2E・サンプリング突合・DEFERREDゼロ）が揃っていなければならない——人間はコードではなく検証報告を見てマージを判断するため、報告なしのマージ要求はゲートを無効化する。完全検証に実APIコスト（レート予算）がかかることは後回しの理由にならない。

## 停止条件

- **正常終了**：対象EPICの「ゴール（ループ停止条件）」充足＝EPIC配下の全チケットが「レビュー中」以上＋**VerifierのEPIC受け入れ検証（完全モード）で受け入れ基準を全件PASS**（最終PRマージ前・対象ブランチ上）。PASS報告をEPICにコメントして終了する（**EPICのステータス遷移はしない**。レビュー中への遷移は報告を確認した人間が行う。子チケットの「完了」遷移は人間のマージ後に行う）。自己判断による「受け入れ基準の充足」はループの終了理由にならない。Verifier未実行・FAIL・NEEDS-HUMANのいずれの状態でも正常終了とみなさない
- **チケット保留（deferred）**：同一チケットで3回連続失敗 → **ループは止めない**。チケットに `deferred` ラベルと保留コメント（試行内容・行き詰まりの理由・考えられる選択肢）を残して保留し、次のチケットへ進む。保留チケットにはループから再着手しない（人間の差し戻しコメント・ラベル解除が再開の合図）
- **保留停止**：未消化チケットが尽き、`deferred` 保留チケットが残っている場合は正常終了できない → EPICに保留チケットの一覧・理由・選択肢をコメントして停止（人間の判断待ち）
- **仕様齟齬停止**：`spec/` とEPICの受け入れ基準に矛盾を発見 → 推測せず `needs-human` で停止
- **予算停止**：Claude Max x5プランのリミット到達 → 進捗をチケットにコメントして停止

## 再開プロトコル

再開時は必ずこの順で状態を復元する：
1. 対象EPICと配下チケットの最新状態・コメントを読む
2. 自分の前回の進捗コメントと未マージのPRを確認する
3. `git status` / ブランチ状態を確認してから作業を続ける
（記憶はコンテキストではなくJIRA/Gitにある。前回の会話内容を前提にしない）

## モデル・コスト方針

- **通常はOpus**を使用する
- **原因不明の不具合・同一問題での行き詰まり時はFableに1ランク上げる**（上げた理由と結果をチケットにコメントする。解消したらOpusに戻す）
- 単純作業（定型コード・ドキュメント整形・コミットメッセージ）はSonnetに下げてよい
- リミット残量を意識し、モデルランクの上げっぱなしにしない

## 禁止事項

- **EPICのステータス遷移**（すべての遷移。進行中への遷移はループ起動トリガーであり、エージェントが触ると自己発火する。EPICへはコメントと `needs-human` ラベルのみ可）
- **PRのマージ**（セルフマージ・auto-merge設定・マージの人間への催促を含む）。マージ権限は人間のみ
- バグ発見時のその場修正（起票のみ）
- `verification/`・`.claude/agents/`・`spec/`・CODEOWNERS・CIワークフローの変更
- EPICの受け入れ基準・ゴールの変更（変更が必要と思う場合は `needs-human` で提案）
