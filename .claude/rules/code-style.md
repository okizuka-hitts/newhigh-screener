# コーディングスタイルガイド

## 基本

- Python 3.12+。フォーマット・Lintはruff（line-length 100、設定は `pyproject.toml` が正）
- 公開関数・メソッドには型ヒント必須。`Any` の安易な使用を避ける
- docstringは日本語でよい。「何を・なぜ」を書く（「どうやって」はコードで語る）

## 構成

```
src/screener/
├── cli.py        # エントリポイント（薄く保つ。ロジックを書かない）
├── api/          # J-Quantsクライアント（認証・レートリミッタ・注入フック）
├── db/           # SQLiteアクセス（スキーマ・クエリ）
├── detect/       # 新高値判定ロジック
└── tune/         # バックテスト・パラメータ調整
```

- モジュール間の依存は上から下へ一方向（cli → detect/tune → db/api）。循環依存禁止
- 設定値・定数（レート上限、DBパス等）はハードコードせず1か所に集約する

## エラーとexit code

- exit code規約：`0`＝成功 / `1`＝データ・実行時エラー / `2`＝使い方の誤り・未実装
- エラーメッセージは「何が起きたか＋ユーザーが次に何をすべきか」を1行で。スタックトレースを既定で見せない
- 例外は握りつぶさない。回復できない場合は明確なメッセージで終了する

## PR・コミット

- 小さく保つ：1PR＝1チケット。無関係な変更を混ぜない
- コミットメッセージ：`feat|fix|test|docs|refactor: 要約 (チケットキー)` 形式。対象チケットのキーを必ず含める
- PR説明には対象チケット・変更概要・テスト方法・Verifierの判定結果を書く
- **マージは人間のみ。** エージェントは自分のPRをマージしない（branch protectionでも強制される）
