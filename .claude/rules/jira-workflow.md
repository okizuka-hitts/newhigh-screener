# JIRAワークフロー

単一の真実はJIRAのNSプロジェクト（hittslabs.atlassian.net）。作業の状態はすべてチケットに残す。

## 役割分担（重要）

- **人間が作るのはEPICだけ**（ゴール・受け入れ基準（Acceptance Criteria）・検証方法を含む）。EPICは変更しない・待たない・催促しない
- **ストーリー・タスク・バグチケットはすべてエージェントが起票する。** 人間の起票を待たずに、EPICの受け入れ基準から必要なストーリーを自分で切り出すこと。ストーリーの分割粒度・順序はあなたの裁量
- Verifierも検証で発見した欠陥のバグチケットを自ら起票する
- 例外は `needs-human` 対応のみ：人間はエスカレーションへの回答とEPICの受け入れ判断だけを行う

## チケットの流れ

- 作業単位は必ずチケット（EPIC直下のストーリー/タスク/バグ）。チケットなしの作業をしない
- 着手時：`未着手 → 進行中` に自分で遷移し、着手コメント（何を・どう進めるか）を残す
- PR作成時：チケットにPRのURLをコメントし、ストーリー検証（軽量モード）PASS後に `進行中 → レビュー中` へ遷移する（人間がレビュー待ちをボードで拾えるようにする）
- **完了（Done）への遷移は実装ループが自分では行わない。** 完了判定は2段階：
  - **ストーリー/タスク/バグ**：Verifierの**ストーリー検証（軽量モード）**でPASS＋**人間がPRをレビューしてマージ** → 両方の充足を確認したうえでエージェントがDoneへ遷移する（マージ前にDoneにしない）
  - **EPIC**：Verifierの**EPIC受け入れ検証（完全モード）**で受け入れ基準全件PASS＋人間の受け入れ判断 → Done

## 起票ルール

- ループの起動は**EPICが「進行中」に遷移したとき、またはRoutineのAPI呼び出し**。起動したら、まず受け入れ基準を読んでストーリー一式を起票してから実装に入る（着手後の追加起票も可）
- ストーリー・タスクは対象EPICの子として起票する。親のないチケットを作らない
- バグは発見時に該当EPIC配下へ起票するだけ。その場で修正しない（割り込み禁止）
- 1チケット＝1事象。まとめ起票しない
- 起票時の必須項目：事象/目的、再現手順または作業内容、**受け入れ条件**（そのチケット固有の完了判定。親EPICのどの受け入れ基準に対応するかも明記）

## ラベル規約

- `needs-human`：人間の判断が必要（3回失敗・仕様齟齬・受け入れ基準の曖昧さ）
- `verifier`：Verifierが起票したバグ
- `test-quality`：テスト品質の指摘（FAIL要因ではない）

## コメント規約

- 作業ログは「やったこと／わかったこと／次にやること」の3点で簡潔に
- 中断時（リミット到達等）は必ず進捗コメントを残してから止まる
- 判断に迷った内容は推測で進めず、チケットに選択肢を書いて `needs-human`
